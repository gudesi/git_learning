# Institutional FX CTA Core Domain Design

Version: 1.0

Status: Approved Baseline

---

# 1. Purpose

This document defines the core domain models of the trading system.

These models form the data contract between all modules.

Any modification to these models may impact multiple layers of the system and therefore must be carefully reviewed.

The goal is to:

* Establish a common language.
* Reduce coupling.
* Stabilize interfaces.
* Minimize future refactoring.

---

# 2. Domain Model Principles

All domain models should follow the following rules:

1. Explicit fields only.
2. No hidden state.
3. Serializable.
4. Type annotated.
5. Business meaning is clear.
6. Infrastructure independent.
7. Broker independent.
8. Backtest and live trading compatible.

---

# 3. Instrument

Purpose:

Represents a tradeable FX instrument.

Examples:

```text
EURUSD
GBPJPY
AUDNZD
USDCHF
```

Fields:

| Field          | Type  | Description       |
| -------------- | ----- | ----------------- |
| symbol         | str   | Instrument symbol |
| base_currency  | str   | Base currency     |
| quote_currency | str   | Quote currency    |
| pip_size       | float | Pip value         |
| contract_size  | float | Contract size     |
| active         | bool  | Tradable flag     |

Lifecycle:

```text
Static
```

Created once during system initialization.

---

# 4. Bar

Purpose:

Represents one daily market observation.

Fields:

| Field     | Type     |
| --------- | -------- |
| symbol    | str      |
| timestamp | datetime |
| open      | float    |
| high      | float    |
| low       | float    |
| close     | float    |
| volume    | float    |

Lifecycle:

```text
Immutable
```

Generated by Data Layer.

Consumed by:

* Signals
* Risk
* Backtest

---

# 5. Signal

Purpose:

Represents a directional trading opinion.

Generated by Signal Engine.

Fields:

| Field                  | Type      |
| ---------------------- | --------- |
| symbol                 | str       |
| timestamp              | datetime  |
| direction              | Direction |
| direction_score        | float     |
| momentum_score         | float     |
| quality_score          | float     |
| risk_adjusted_momentum | float     |
| final_score            | float     |

Direction:

```text
LONG
SHORT
FLAT
```

Lifecycle:

```text
Daily
```

Generated once per rebalance cycle.

Consumed by:

* Candidate Pool
* Portfolio Builder

| Field          | Type      |

# 8. Order

Represents an executable instruction.

Fields:

| Field      | Type         |
| ---------- | ------------ |
| order_id   | str          |
| symbol     | str          |
| side       | OrderSide    |
| quantity   | float        |
| order_type | OrderType    |
| price      | float | None |
| stop_price | float | None |
| created_at | datetime     |
| status     | OrderStatus  |

Order Side:

```text
BUY
SELL
```

Order Type:

```text
MARKET
LIMIT
STOP
```

Order Status:

```text
PENDING
FILLED
CANCELLED
REJECTED
```

Lifecycle:

```text
Created
↓
Submitted
↓
Filled
```

or

```text
Created
↓
Cancelled
```

# 9. PortfolioState

Purpose:

Represents the entire portfolio at a point in time.

Fields:

| Field          | Type           |
| -------------- | -------------- |
| timestamp      | datetime       |
| equity         | float          |
| cash           | float          |
| leverage       | float          |
| positions      | list[Position] |
| realized_pnl   | float          |
| unrealized_pnl | float          |
| drawdown       | float          |

Lifecycle:

```text
Continuous
```

Updated every rebalance cycle.

Consumed by:

* Risk Engine
* Monitoring
* Backtest

| Field           | Type        |
| --------------- | ----------- |
| timestamp       | datetime    |

| target_leverage | float       |
| expected_risk   | float       |

Lifecycle:

```text
Daily
```

Generated after Risk Engine approval.

Consumed by:

* Execution Layer

---

# 12. Data Flow

The primary business flow is:

```text
Bar
↓
Signal
↓
Candidate
↓
PortfolioState (Target)
↓
RiskReport
↓
ExecutionPlan
↓
Order
↓
PortfolioState (Actual)
```

Every module must consume and produce these domain objects whenever possible.

Primitive values should not be passed between modules if a domain object already exists.

---

# 13. Ownership Rules

The following ownership hierarchy applies:

```text
PortfolioState
│
├── Position
│
└── RiskReport
```

```text
ExecutionPlan
│
└── Order
```

Signals do not own Positions.

Orders do not own Positions.

Positions are owned only by PortfolioState.

---

# 14. Modification Policy

Changes to the following models require architecture review:

```text
Instrument
Signal
Position
Order
PortfolioState
RiskReport
```

These are considered system-critical domain models.

---

# 15. Future Extensions

The architecture should support future additions without breaking existing interfaces.

Potential future additions:

* Multi-asset support
* Futures support
* Options support
* Multiple brokers
* Intraday execution
* Machine learning overlays

The current design should remain backward compatible.
